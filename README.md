# tg (Tool Gateway)

Расширяемая платформа, которая выступает в роли **планировщика и исполнителя** для ваших приложений и инструментов. Она позволяет выстраивать сложные рабочие процессы в виде **цепочек из плагинов**, упрощая их доставку, версионирование и управление зависимостями.

Платформа не ограничена какой-либо конкретной областью применения: на `tg` можно создавать любые приложения — от утилит для разработки и автоматизации до менеджеров паролей, криптокошельков, генераторов TOTP-кодов и других специализированных инструментов. Каждый плагин представляет собой полноценное приложение с собственной бизнес-логикой, которое может быть интегрировано в общий рабочий процесс.

## Целевая аудитория

`tg` предназначен для:

- **DevOps-инженеров и SRE**, которым необходимо предоставлять командам стандартизированные и надежные инструменты для управления инфраструктурой и CI/CD.
- **Технических лидеров и архитекторов**, которые занимаются унификацией и композицией процессов автоматизации.
- **Разработчиков**, которым нужен единый интерфейс для выполнения рутинных задач без необходимости вникать в детали реализации каждого инструмента.
- **Создателей специализированных приложений**, которые хотят разрабатывать безопасные и изолированные инструменты с возможностью их простого распространения и интеграции в общие рабочие процессы.

## Решаемые задачи

В проектах часто используется множество разрозненных утилит и инструментов, что создает следующие сложности:

- **Отсутствие единого интерфейса:** Разные инструменты имеют разные CLI, форматы конфигурации и способы запуска, что усложняет их использование и обучение команды.
- **Сложность композиции инструментов:** Объединение нескольких инструментов в единый рабочий процесс требует написания оберток и скриптов, что усложняет поддержку и делает процессы хрупкими.
- **Отсутствие механизма переиспользования сквозной функциональности:** Авторизация, логирование, телеметрия и другие сквозные функции приходится встраивать в каждый инструмент отдельно, что приводит к дублированию кода.
- **Проблемы с версионированием и зависимостями:** Отсутствие централизованного управления версиями инструментов затрудняет отслеживание совместимости и обновление инструментов в команде.
- **Конфликты окружений:** Разные проекты могут требовать разных версий одних и тех же инструментов, что создает конфликты при работе над несколькими проектами одновременно.
- **Сложность распространения:** Отсутствие единого механизма доставки инструментов команде приводит к ручному копированию, настройке окружений и рассинхронизации версий между разработчиками и CI/CD.
- **Безопасность выполнения:** Выполнение стороннего кода напрямую создает риски безопасности, особенно при использовании инструментов из внешних источников.

`tg` помогает решить эти проблемы, предоставляя структурированную среду для композиции, управления и исполнения инструментов. Платформа может быть использована для организации и унификации процессов автоматизации в команде, а также для создания и распространения любых специализированных приложений — от утилит разработки до менеджеров паролей и криптографических инструментов.

## Установка

`tg` можно установить несколькими способами:

### Установка через Homebrew

Рекомендуемый способ установки для пользователей macOS и Linux:

```bash
brew tap seniorGolang/cli
brew install tg
```

### Установка через go install

Для установки из исходного кода используйте:

```bash
go install github.com/seniorGolang/tg/v3/cmd/tg@latest
```

После установки хоста можно активировать встроенные agent skills:

```bash
tg skills install
# или с созданием каталога, если ещё нет ~/.agents:
tg skills install --skills-mkdir
```

## Ключевые возможности

- **Встроенный планировщик исполнения и композитные команды:**

  - `tg` оснащён встроенным планировщиком, который автоматически строит и управляет цепочками плагинов для выполнения команд.
  - Планировщик не просто запускает один плагин, а автоматически формирует оптимальную последовательность выполнения: собирает зависимости, разрешает версии, проверяет совместимость и строит план выполнения в правильном порядке (pre-run -> stage -> command -> post-run).
  - Это позволяет создавать гибкие рабочие процессы без необходимости ручного управления зависимостями и порядком выполнения. Например, можно легко встроить плагин авторизации в начало цепочки для всех команд, а плагин для отправки телеметрии — в конец, и планировщик автоматически включит их в план выполнения.
  - Разные команды могут использовать разные цепочки исполнения, добавляя или убирая шаги валидации и нотификации, но переиспользуя при этом основные плагины. Планировщик автоматически адаптирует план под каждую команду.

- **Полный набор инструментов для разработки плагинов:**

  - `tg` предоставляет полный набор инструментов для создания, разработки и деплоя плагинов.
  - Команда `tg plugin init` автоматически генерирует базовую структуру плагина, включая шаблоны кода, конфигурационные файлы и документацию.
  - Автоматическая генерация CI/CD конфигураций для деплоя плагинов в GitHub и GitLab репозитории, которые включают процесс сборки и публикации плагинов.
  - Поддержка создания плагинов на различных языках программирования, что снимает технологические ограничения.

- **Доставка инструментов и управление зависимостями:**

  - Встроенный менеджер пакетов (`tg pkg add`, `tg pkg upgrade`) упрощает установку и обновление плагинов.
  - `tg` автоматически управляет зависимостями между плагинами, разрешая конфликты и обеспечивая консистентность.

- **Изолированные окружения (Scopes):**

  - Позволяют создавать отдельные конфигурации для разных проектов, гарантируя консистентность окружения у всех участников команды.

- **Безопасное выполнение с полным контролем над средой:**
  - `tg` обеспечивает полный контроль над средой выполнения плагинов, включая все аспекты взаимодействия с системой.
  - **Контроль файловой системы:** Каждый плагин может работать только с явно разрешёнными путями и директориями, с указанием уровня доступа (чтение или запись). Доступ к остальным файлам системы полностью заблокирован.
  - **Контроль сетевых подключений:** Плагины могут устанавливать соединения только с хостами из предварительно утверждённого списка, что предотвращает несанкционированный доступ к внешним ресурсам.
  - **Контроль переменных окружения:** Плагины получают доступ только к тем переменным окружения, которые явно указаны в их конфигурации, защищая конфиденциальные данные системы.
  - **Контроль выполнения команд:** Плагины могут запускать только те системные команды, которые явно разрешены, что исключает выполнение произвольного кода на хосте.
  - **Изоляция выполнения:** Плагины выполняются в полностью изолированной среде, что исключает влияние на основную систему и другие плагины.

## Примеры

Для изучения практических примеров использования `tg` и разработки плагинов, ознакомьтесь со следующими проектами:

### tgp-go

Набор плагинов для автоматической генерации кода на основе Go интерфейсов с аннотациями `@tg`.

Предоставляет инструменты для создания полноценных серверных и клиентских приложений из описания контрактов, поддерживая HTTP REST API и JSON-RPC 2.0 протоколы.

**Включает плагины:**

- `astg` — парсер проекта и аннотаций
- `server` — генератор серверного кода на основе Fiber
- `client-go` — генератор типобезопасного Go клиента
- `client-ts` — генератор TypeScript клиента
- `swagger` — генератор OpenAPI 3.0 документации

**Установка:**

```bash
tg pkg add https://github.com/seniorGolang/tgp-go
```

**Документация:**

```bash
tg plugin doc astg
tg plugin doc server
tg plugin doc client-go
tg plugin doc client-ts
tg plugin doc swagger
```

**Репозиторий:** [tgp-go](https://github.com/seniorGolang/tgp-go)
